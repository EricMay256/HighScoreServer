# 0002. Raw SQL over ORM

Date: 2026-04-12

## Status

Accepted

## Context

The leaderboard's core queries lean on PostgreSQL features that resist ORM abstraction: window functions (`RANK() OVER (...)`, `COUNT(*) OVER ()`) for rank and percentile, period-bucketed upserts using `INSERT ... ON CONFLICT ... DO UPDATE WHERE` with a conditional that depends on the game mode's sort direction, and dynamic `ORDER BY` direction selected per query. SQLAlchemy can express these, but only by dropping into `text()` fragments, at which point the ORM is providing connection management and not much else.

The argument for an ORM at this scale is usually migrations: SQLAlchemy plus Alembic gives you autogenerated schema diffs, which is genuinely valuable on a team. Autogeneration requires the ORM to be aware of the model classes, which would mean defining duplicate model classes purely to feed Alembic, while the actual queries continued to use raw SQL.

## Decision

Use psycopg2 directly with raw SQL strings. Manage schema via `db/schema.sql` applied manually (locally) or via `heroku pg:psql` (production). Do not introduce SQLAlchemy or Alembic.

## Consequences

Every query is greppable, copy-pastable into `psql` for debugging, and visible in the same file as the route handler that issues it. The query logic is not hidden behind a builder, which makes it easy to reason about exactly what hits the database on any given request.

The cost is no compile-time safety on column names or types — a typo in a column reference is a runtime error, caught by tests rather than by tooling. Schema changes require touching every query that references the changed column by hand, since there is no model class to refactor through. At the current query count this is tractable; at 10x the query surface it would start to hurt.

This decision is coupled to two others. ADR 0005 (sync over async) is enabled by it: psycopg2 is synchronous, and an ORM with async support would have made the sync/async question more interesting. ADR 0003 (period bucketing via upsert) depends on it: the conditional upsert pattern is a SQL feature, not an ORM feature, and would have required `text()` regardless. If either of those decisions is later revisited, this one should be revisited at the same time.

The migration path, if it ever becomes necessary, is psycopg2 → asyncpg, keeping raw SQL throughout. SQLAlchemy and Alembic are not on that path.
